### 4. Use Case (Варианты использования)
*Описывают пошаговое взаимодействие клиента с **мобильным приложением** (не с бэкендом!).*

#### UC-01: Успешное бронирование тренировки
* **ID:** UC-01
* **Формулировка:** Клиент выбирает слот, указывает опцию проката и успешно записывается на тренировку.
* **Приоритет:** Высокий
* **Актор:** Клиент
* **Предусловие:** Клиент авторизован, бэкенд отдал список доступных слотов.
* **Основной поток:**
  1. Клиент выбирает слот из списка (в пределах 7 дней или через фильтр).
  2. Приложение отображает экран подтверждения с выбором опции снаряжения (свое/прокат).
  3. Клиент подтверждает выбор.
  4. Приложение отправляет запрос на бронирование в API.
  5. Приложение получает от бэкенда ответ «Успешно» и отображает экран активной брони.
* **Постусловие:** Бронь создана, статус отображается как «Активна».
* **Источник:** [Бриф], [R-004], [Домен]

#### UC-02: Обработка отказа в бронировании (Нет мест / Нет снаряжения)
* **ID:** UC-02
* **Формулировка:** Приложение корректно обрабатывает и отображает отказ бэкенда в создании брони.
* **Приоритет:** Высокий
* **Актор:** Клиент
* **Предусловие:** Клиент инициировал процесс бронирования.
* **Основной поток:**
  1. Клиент нажимает кнопку подтверждения записи.
  2. Приложение отправляет запрос в API.
  3. Бэкенд возвращает ошибку (например, место занял другой клиент в ту же секунду или закончился фонд проката).
  4. Приложение перехватывает ошибку и отображает клиенту сообщение о невозможности записи (текст ошибки формируется на основе кода ответа API).
  5. Клиент остается на экране выбора слотов.
* **Постусловие:** Бронь не создана, состояние UI сброшено.
* **Источник:** [R-004], [Домен]

#### UC-03: Отмена записи клиентом
* **ID:** UC-03
* **Формулировка:** Клиент отменяет свою активную бронь через интерфейс приложения.
* **Приоритет:** Высокий
* **Актор:** Клиент
* **Предусловие:** У клиента есть бронь со статусом «Активна».
* **Основной поток:**
  1. Клиент открывает детали брони и нажимает кнопку «Отменить».
  2. Приложение отправляет запрос на отмену в API.
  3. Приложение получает подтверждение от бэкенда.
  4. Приложение обновляет UI, меняя статус брони на «Отменена клиентом».
* **Постусловие:** Бронь переведена в статус «Отменена клиентом».
* **Источник:** [Бриф], [Домен]

#### UC-04: Отображение отмены тренировки скалодромом
* **ID:** UC-04
* **Формулировка:** Приложение отображает факт отмены слота организаторами и блокирует повторную запись.
* **Приоритет:** Высокий
* **Актор:** Клиент
* **Предусловие:** Бэкенд инициировал отмену слота (профилактика).
* **Основной поток:**
  1. Приложение получает от бэкенда обновление статуса слота (через push-уведомление или при следующем запросе расписания).
  2. Приложение изменяет статус связанной брони клиента на «Отменена скалодромом».
  3. Приложение отображает причину отмены (полученную от бэкенда).
  4. Приложение делает неактивной (скрытой или заблокированной) возможность повторной записи на этот конкретный слот.
* **Постусловие:** Клиент уведомлен, повторная запись на слот заблокирована на уровне UI.
* **Источник:** [R-008], [Домен]

#### UC-05: Оценка инструктора
* **ID:** UC-05
* **Формулировка:** Клиент отправляет оценку инструктору по факту состоявшейся тренировки.
* **Приоритет:** Средний
* **Актор:** Клиент
* **Предусловие:** Время окончания тренировки, на которую был записан клиент, наступило.
* **Основной поток:**
  1. Клиент открывает интерфейс оценки (через историю или уведомление).
  2. Клиент вводит оценку (формат оценки определяется контрактом API).
  3. Клиент нажимает «Отправить».
  4. Приложение отправляет данные отзыва в API и отображает сообщение об успешной отправке.
* **Постусловие:** Отзыв отправлен на бэкенд.
* **Источник:** [Бриф], [Домен]
