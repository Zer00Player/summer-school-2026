
### 2. Нефункциональные требования (НФТ)
*Описывают архитектурные ограничения, качества системы (usability, reliability) и технологические границы.*

| ID | Формулировка требования | Приоритет | Источник |
| :--- | :--- | :--- | :--- |
| **НФТ-01** | **Usability (Юзабилити):** Интерфейс приложения должен быть спроектирован по принципу Mobile-first, так как целевое использование происходит со смартфонов в зале. | Высокий | [Бриф] |
| **НФТ-02** | **Architecture (Архитектура):** Приложение не должно дублировать на клиенте бизнес-логику проверки доступности мест и снаряжения. Гарантия атомарности (защита от двойных броней) обеспечивается исключительно доверием к ответам API бэкенда (Black-box). | Высокий | [R-004], [Домен] |
| **НФТ-03** | **Data (Данные):** Приложение должно работать исключительно с данными, предоставляемыми через канонический контракт API в реальном времени. Механизмы локального кэширования исторических данных или миграции легаси-схем не предусмотрены. | Высокий | [R-015], [Домен] |
| **НФТ-04** | **Constraint (Ограничение скоупа):** Архитектура и UI приложения должны строго ограничиваться ролью «Клиент». Эндпоинты и сценарии для ролей «Инструктор» и «Администратор» не должны быть доступны или реализованы в клиентской сборке. | Высокий | [R-028] |
| **НФТ-05** | **Constraint (Платежи):** В текущей версии приложения должны отсутствовать модули, формы и интеграции для обработки онлайн-оплат (эквайринг). | Низкий | [Бриф] |
