# Дизайн-Брифинг: SCR-04 - Экран активной брони

## 1. Общая информация
- **ID экрана:** SCR-04
- **Название:** Экран активной брони
- **Приоритет:** Высокий
- **Связанные требования:** US-04, US-05, ФТ-07, ФТ-08, ФТ-09

## 2. Цель экрана
Предоставить клиенту полную информацию о его активной брони и возможность отменить её в случае изменения планов, а также увидеть критическую информацию об отмене со стороны скалодрома.

## 3. Целевая аудитория
- Клиенты с активной записью на тренировку (в ближайшие часы/дни)
- Могут открывать экран для: проверки деталей, отмены, просмотра уведомлений об отмене

## 4. Ключевые пользовательские сценарии
1. **Первичный сценарий:** Пользователь открывает свою активную бронь → видит детали → при необходимости нажимает "Отменить" → подтверждает отмену → статус меняется
2. **Сценарий отмены скалодромом:** Пользователь получает push-уведомление → открывает бронь → видит статус "Отменена скалодромом" и причину → кнопка отмены неактивна
3. **Edge case:** Попытка отмены в недопустимое время (за 10 минут до тренировки) - приложение должно показать предупреждение (сообщение от API)

## 5. Требования к контенту и данным
- **Информация о брони:** Дата, время, тип тренировки, инструктор (если доступен)
- **Опция снаряжения:** "Свое снаряжение" или "Прокат"
- **Статус брони:** 
  - "Активна" (по умолчанию)
  - "Отменена клиентом"
  - "Отменена скалодромом" (с причиной)
- **Причина отмены скалодромом:** Текст от бэкенда (профилактика, закрытие зоны и т.д.)
- **Кнопка отмены:** Доступна только для активной брони (не для отмененных)

## 6. Функциональные элементы UI
- **Заголовок экрана:** "Моя бронь" или название тренировки
- **Карточка деталей:** Полная информация о брони
- **Статусный баджик:** Визуальное обозначение статуса (цветной ярлык)
- **Кнопка отмены:** "Отменить запись" (красный цвет, заметная) 
- **Кнопка "Назад":** Возврат на главный экран
- **Push-уведомление:** Если бронь отменена скалодромом - визуальное выделение причины
- **Информационный блок:** Правила отмены (если есть - лимит по времени)

## 7. Визуальные требования
- **Четкая иерархия:** Статус - первое, что видит пользователь
- **Баланс:** Информация не перегружена, но полна
- **Акцент на статусе:** Цвет баджика соответствует статусу (зеленый - активна, серый - отменена, красный - отменена скалодромом)
- **Крупные элементы:** Кнопка отмены не менее 44px, легко нажимается
- **Информативность:** Причина отмены скалодромом выделена визуально (отдельный блок или иконка)

## 8. Состояния экрана
| Состояние | Статус | Визуальное представление |
|:---|:---|:---|
| **Активна** | "Активна" | Зеленый баджик, кнопка отмены активна |
| **Отменена клиентом** | "Отменена" | Серый баджик, кнопка отмены скрыта/неактивна |
| **Отменена скалодромом** | "Отменена скалодромом" | Красный/оранжевый баджик, блок с причиной, кнопка отмены неактивна |

## 9. Взаимодействия и анимации
- **Отмена брони:** 
  1. Нажатие на "Отменить"
  2. Показ confirm-диалога: "Вы уверены, что хотите отменить запись?"
  3. Отправка запроса в API
  4. Успех → обновление статуса на "Отменена клиентом"
  5. Ошибка → показ сообщения об ошибке
- **Обновление статуса:** При получении push-уведомления автоматически обновляется UI
- **Переходы:** Возврат на главный экран через "Назад"

## 10. Варианты дизайна (для рассмотрения)
1. **Вариант A (Карточка деталей):** Информация в виде карточки с акцентами, баджиком статуса и кнопкой внизу
2. **Вариант B (Разделенный экран):** Верхняя часть - информация о брони, нижняя - правила и кнопка отмены
3. **Вариант C (Список + Детали):** Список активных броней (если их несколько) с возможностью открыть детали

**Рекомендуемый вариант:** Вариант A (Карточка деталей) - просто и понятно для одного активного бронирования.

## 11. Ограничения и технические нюансы
- **Бизнес-логика отмены:** Валидация времени отмены (за 10 минут) на бэкенде - UI показывает результат (успех/ошибка)
- **Атомарность:** Защита от двойной отмены на бэкенде
- **Push-уведомления:** При открытии экрана после push - актуальный статус загружается из API
- **Блокировка повторной записи:** Если слот отменен скалодромом, повторная запись на него недоступна (UI блокирует на SCR-01)

## 13. Критерии приемки дизайна
- [ ] Все детали брони отображаются корректно
- [ ] Статус брони четко виден и цветовое кодирование соответствует статусу
- [ ] Причина отмены скалодромом отображается в отдельном блоке
- [ ] Кнопка отмены доступна только для активной брони
- [ ] При попытке отмены показывается confirm-диалог
- [ ] При успешной отмене статус обновляется на "Отменена клиентом"
- [ ] При ошибке отмены показывается сообщение об ошибке
- [ ] Кнопка "Назад" возвращает на главный экран
