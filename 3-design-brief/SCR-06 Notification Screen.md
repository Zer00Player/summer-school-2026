# Дизайн-Брифинг: SCR-06 - Экран уведомлений

## 1. Общая информация
- **ID экрана:** SCR-06
- **Название:** Экран уведомлений
- **Приоритет:** Высокий
- **Связанные требования:** US-06, US-05, ФТ-11

## 2. Цель экрана
Централизовать все push-уведомления, относящиеся к тренировкам (напоминания, отмены), чтобы клиент мог просмотреть их в любое время и не пропустить важную информацию.

## 3. Целевая аудитория
- Клиенты, получающие push-уведомления от приложения
- Могут открывать экран для просмотра истории уведомлений или непрочитанных

## 4. Ключевые пользовательские сценарии
1. **Первичный сценарий:** Пользователь получает push-уведомление о тренировке → открывает приложение → видит уведомление в центре
2. **Сценарий отмены:** Пользователь получает push-уведомление об отмене → открывает центр уведомлений → видит причину отмены → переходит к брони
3. **Edge case:** Уведомления почищены системой - центр пуст

## 5. Требования к контенту и данным
- **Типы уведомлений:**
  - Напоминание о предстоящей тренировке (за 2 часа)
  - Отмена тренировки скалодромом (с причиной)
  - Изменение статуса брони
  - Возможно: информация о новых тренировках
- **Данные уведомления:** Заголовок, текст, дата/время, иконка/тип, связанный ID брони (для перехода)
- **Источник данных:** Push-уведомления от бэкенда (FCM/APNS)

## 6. Функциональные элементы UI
- **Заголовок экрана:** "Уведомления"
- **Список уведомлений:** От новых к старым, сгруппированы по датам
- **Карточка уведомления:** Иконка типа, заголовок, текст, время
- **Клик на карточку:** Переход к соответствующей брони (если применимо)
- **Кнопка "Очистить все":** Опционально - для удаления всех уведомлений
- **Индикатор непрочитанных:** Жирный шрифт или точка для непрочитанных
- **Empty state:** "Нет уведомлений" с иллюстрацией

## 7. Визуальные требования
- **Иерархия:** Самые новые - сверху
- **Группировка:** По датам (Сегодня, Вчера, Ранее...)
- **Типизация:** Разные иконки для разных типов уведомлений (календарь, отмена, напоминание)
- **Четкость:** Текст уведомлений должен быть легко читаемым

## 8. Состояния экрана
| Состояние | Описание | Визуальное представление |
|:---|:---|:---|
| **Загрузка** | Данные загружаются | Скелетон-загрузка |
| **Есть уведомления** | Список получен | Отображение карточек |
| **Нет уведомлений** | Пустой центр | Empty state |
| **Ошибка** | Ошибка загрузки | Сообщение с кнопкой "Повторить" |

## 9. Взаимодействия и анимации
- **Pull-to-refresh:** Обновление списка
- **Клик на уведомление:** Переход к связанному экрану (SCR-04 или SCR-05)
- **Маркировка прочитанного:** При клике уведомление становится прочитанным
- **Плавный скролл:** Бесконечный скролл

## 10. Варианты дизайна (для рассмотрения)
1. **Вариант A (Простой список):** Классический список уведомлений с иконками и текстом
2. **Вариант B (Группировка по типу):** Группировка уведомлений по типам (Напоминания, Отмены)
3. **Вариант C (С баджами):** Индикатор непрочитанных в виде красного баджа

**Рекомендуемый вариант:** Вариант A (Простой список) + Вариант C (баджи) - стандартный и понятный подход.

## 11. Ограничения и технические нюансы
- **Источник данных:** Push-уведомления от бэкенда
- **Локальное хранение:** Уведомления могут храниться локально (последние N штук) или получаться из API
- **Переходы:** Клик на уведомление открывает соответствующую бронь (если применимо)
- **Удаление:** Пользователь может очистить уведомления вручную

## 12. Референсы и вдохновение
- Приложения: Instagram (уведомления), Telegram, ВКонтакте
- Операционные системы: iOS Notification Center, Android Notifications

## 13. Критерии приемки дизайна
- [ ] Все уведомления отображаются с корректным типом (иконка)
- [ ] Уведомления сгруппированы по датам
- [ ] Непрочитанные уведомления визуально отличаются (жирный шрифт, точка)
- [ ] При клике на уведомление осуществляется переход к связанной брони
- [ ] Empty state появляется при отсутствии уведомлений
- [ ] Кнопка "Очистить все" работает корректно (если реализована)
